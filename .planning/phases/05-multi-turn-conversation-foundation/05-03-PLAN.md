  <name>Task 1: Add --resume CLI flag and session lifecycle</name>
  <files>src/claude_teletype/cli.py, src/claude_teletype/tui.py</files>
  <action>
    **CLI --resume flag (cli.py):**
    - Add `resume: str = typer.Option(None, "--resume", help="Resume a previous session by ID")` to the chat() command parameters
    - Pass `resume_session_id=resume` to TeletypeApp constructor
    - After `tui_app.run()` returns, check `tui_app.session_id` (a new property). If it has a value, print: `"To resume: claude-teletype --resume {session_id}"`

    **TUI resume support (tui.py):**
    - Add `resume_session_id: str | None = None` parameter to TeletypeApp.__init__
    - Store as `self._session_id = resume_session_id` (replaces the default None)
    - Add a `session_id` property that returns `self._session_id` (for CLI to read after exit)
    - In on_mount, if `self._session_id` is not None, write a brief resume message to the log: `"Resumed session {session_id[:8]}...\n\n"` and update the status bar placeholder text
    - The first prompt submission will use --resume with the provided session_id automatically (already handled by stream_response passing self._session_id)

    **Session resume failure handling (tui.py stream_response):**
    - If StreamResult.is_error is True after a --resume attempt: reset self._session_id to None, write warning to log: `"\n[Session expired or corrupted. Starting new session.]\n"`, and the next prompt will start fresh (no --resume)
    - Per user decision: do NOT retry --resume on the same session_id after failure

    **Exit session_id display (cli.py):**
    - After tui_app.run(), print session_id to stderr (not stdout) so it does not interfere with piped output
    - Use `console.print(f"To resume: claude-teletype --resume {tui_app.session_id}", style="dim")` for styled output
