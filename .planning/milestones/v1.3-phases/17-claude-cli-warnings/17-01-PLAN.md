    - No changes needed to the SettingsScreen itself -- it returns the dict of new values. The confirmation happens in TUI's `_apply_settings` callback.

    **TUI hot-swap confirmation** (`tui.py` in `_apply_settings`):
    - Before the backend change block (lines 249-269), detect if the backend is actually changing: `result["backend"] != self._backend_name`.
    - If changing, show a confirmation dialog. Use Textual's built-in approach: refactor the backend-change logic into a separate async method `_confirm_and_swap_backend(result: dict)` that:
      1. Creates an inline confirmation using `self.push_screen()` with a simple confirmation screen, OR uses Textual's `self.app.push_screen()` with an `InformationScreen`/custom confirmation.
      2. **Simplest approach**: Use a small `ConfirmSwapScreen(ModalScreen[bool])` class (define it in `tui.py` near `_apply_settings`):
         - Composes a Vertical container with:
           - A warning message Static: "Switching backends will lose your current session context. Conversation history from the claude-cli session cannot be transferred.\n\nContinue?"
           - Two buttons: "Switch Backend" (primary, dismisses True) and "Cancel" (dismisses False)
         - Minimal CSS for centered modal
      3. In `_apply_settings`, when backend changes detected:
         - Call `self.push_screen(ConfirmSwapScreen(), callback=self._handle_swap_confirmation)` passing the result dict via a closure or by storing it temporarily
         - `_handle_swap_confirmation(confirmed: bool)` proceeds with existing backend swap logic if True, does nothing if False
         - The rest of `_apply_settings` (delay, audio, profile, save) still applies immediately -- only the backend swap is gated

    **Alternative simpler pattern** (preferred if Textual supports it):
    - Check if Textual has `action_confirm` or similar. If not, the ConfirmSwapScreen approach above is clean.
    - The ConfirmSwapScreen should be ~30 lines: a ModalScreen[bool] with a warning message and two buttons.

    - Only warn when switching AWAY FROM claude-cli (per CONTEXT.md discretion -- claude-cli is the backend with session state that gets lost). If switching between openai and openrouter, no warning needed (API backends don't have persistent sessions).
    - If user cancels: revert the backend selection, keep current backend, still apply other settings (delay, audio, profile).
    - If user confirms: proceed with existing backend swap logic.
