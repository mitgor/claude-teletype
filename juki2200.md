# Juki 2200 Hardware Integration — Findings & Plan

## Context

The project's primary physical test target is a **1985 Juki 2200 daisywheel
memory typewriter** with a Centronics LPT interface. It's currently connected
via a cheap CH341-based USB-to-LPT adapter (`USB ID 0x1a86:0x7584`, marketed
as "USB2.0-Print"). Goal: stream characters from the `claude-teletype` CLI
to the Juki 2200 with the same character-by-character pacing the project
already does for the Juki 6100.

## Findings (session 2026-04-25 → 2026-04-27)

### Hardware behavior (verified empirically)

- **Mode switch.** The Juki 2200 enters PRINT/online mode on `CODE` + `PRINT`
  key combo. Until that combo is pressed, the typewriter ignores the LPT bus.
- **Default margins.** Power-on margins are left = column 30, right = column
  90 at 10 cpi. The typewriter does NOT auto-wrap — bytes past column 90
  produce a buzzer beep and are discarded. A Diablo HMI reset (`ESC FS NL`,
  `1B 1C 0C`) or Juki full reset (`ESC SUB I`, `1B 1A 49`) at job start, or
  manual margin widening on the front panel, is required before printing
  long lines.
- **Line buffer.** The typewriter is line-buffered, not stream-buffered.
  Bytes accumulate internally and the daisywheel does not strike until a
  `CR LF` (0x0D 0x0A, two separate bytes, in that order) terminator arrives.
  Lone `\n` produces stair-step or no print. Lone `\r` overprints the same
  line. Always emit `b"\r\n"` from binary writes.
- **Initial enumeration garbage.** When the CH341 adapter is plugged in and
  the typewriter is then put into print mode, the typewriter strikes a
  small number of identical characters (mit observed "nu"-symbol-like
  glyphs). This is the chip asserting a byte on D0-D7 during enumeration
  that the typewriter latches once mode-switch pulses /STROBE. Sending
  `SOFT_RESET` or `nINIT` afterwards does not always clear this; the
  cleanest workaround is to enter print mode after the adapter has
  enumerated (i.e. plug USB first, then `CODE`+`PRINT`).

### CH341 USB-LPT adapter (`0x1a86:0x7584`)

- Three alt settings on interface 0:
  - `alt 0`: USB Printer Class (1 OUT bulk endpoint, unidirectional IEEE 1284)
  - `alt 1`: USB Printer Class (OUT + IN bulk endpoints, bidirectional IEEE 1284)
  - `alt 2`: Vendor-specific (CH341 raw parallel mode; needs the chip's
    vendor control transfers to drive — Linux `ch341.ko` is the reference)
- USB Printer Class control queries (`GET_DEVICE_ID`, `GET_PORT_STATUS`,
  `SOFT_RESET`) **STALL with EPIPE** on this chip. It is a thin pass-through,
  not a full Printer-Class implementation.
- In Printer-Class mode (alt 0/1) the chip handles Centronics handshake
  internally and silently drops bytes when /BUSY violations occur — it does
  **not** surface /BUSY back to the host. Drip-feeding bytes from the host
  side (≤10 cps) does not change this; the chip's internal handshake is
  the bottleneck, not USB pacing.
- Net result: bytes that leave pyusb successfully (`OK (N bytes)` in the
  probe table) may never reach the daisywheel.

### macOS print stack interference

- macOS auto-creates a CUPS queue ("USB2.0-Print") on plug-in. The CUPS USB
  backend then exclusively holds the device handle, which makes pyusb's
  `claim_interface(0)` fail with `EACCES` and class-specific control
  transfers fail with `EPIPE`.
- `cupsdisable USB2.0-Print` is NOT enough — the queue stays registered and
  the USB backend keeps the device open.
- Reliable release path, in escalating order:
  1. Physical unplug + replug, then run the probe within ~2 s.
  2. `lpadmin -x USB2.0-Print` to delete the queue entirely.
  3. `sudo launchctl bootout system/org.cups.cupsd` to stop CUPS for the
     session.
  4. `sudo kextunload -b com.apple.driver.AppleUSBPrinter` (SIP-permitting,
     rare on Apple Silicon).

### Profile state (in-tree)

`src/claude_teletype/profiles.py`:

- `juki-6100`: canonical profile (was `juki`); init = `ESC SUB I`, line spacing
  `ESC RS 9`, fixed pitch `ESC Q`, CRLF newlines, reinit-on-newline. Targets
  the older Juki 6100/9100 daisywheel printers.
- `juki-2200`: new entry (this session). Conservative typewriter defaults —
  no init/reset ESC sequences, CR+LF newlines, no formfeed-on-close, no
  VID/PID bound (shares the CH341 adapter with the 6100 so explicit
  selection only). To be refined once a working byte-path is established
  on real hardware.
- `juki`: backward-compat alias resolving to `juki-6100`. Preserves existing
  `--juki` flag and `profile = "juki"` config keys.

## Probe tool — `juki2200.py`

A standalone pyusb script at the repository root for hardware-level
diagnostics. Honors all the Centronics/typewriter gotchas: binary CR+LF
writes, margin-safe line lengths, ESC reset attempts before payloads,
adapter-class diagnostic, per-byte drip-feed mode, and explicit handling
of macOS print-stack interference.

Key invocations:

```bash
# Most spec-clean test, single 'A' on its own line, slow per-byte writes
uv run python juki2200.py --minimal --alt 0 --slow --byte-pause 0.5 --reset

# Same on the bidirectional alt setting
uv run python juki2200.py --minimal --alt 1 --slow --byte-pause 0.5 --reset

# Full bruteforce table across all alts, default pacing
uv run python juki2200.py --reset

# Spec-margin pacing (1 s/byte, lets /BUSY fully release between strikes)
uv run python juki2200.py --slow --byte-pause 1.0 --reset
```

Current outcome on the live setup: writes return `OK` from pyusb but the
daisywheel does not strike. Strong evidence the CH341 in printer-class mode
is buffering host bytes and not flushing them through to the parallel pins.

## Plan — what to try when new hardware arrives

mit is sourcing a new LPT adapter and an Arduino. Proposed paths in
priority order:

### Path A (preferred): real bidirectional USB-LPT adapter

A "true" IEEE 1284 ECP/EPP adapter exposes itself as a character device
(`/dev/usb/lp0` on Linux, or as a CUPS-bypassable raw bulk endpoint on
macOS) and faithfully relays Centronics handshake.

When the new adapter arrives:

1. Plug it in, run `juki2200.py` to inspect interface descriptors and
   adapter-class banner.
2. If `class 0xff` (vendor-specific) endpoints are present on the same alt
   as the printer-class ones, suspect a chip that supports both modes.
3. Run `uv run python juki2200.py --minimal --alt 0 --slow --byte-pause
   0.5 --reset` first; if 'A' strikes, the rest is profile-tuning.
4. If still no strike, try `--alt 1` and `--alt 2`.

### Path B: Arduino as USB-to-Centronics bridge

Use an Arduino (Uno, Pro Micro, or any AVR/RP2040 with ≥13 GPIO) as a
deterministic, host-controlled IEEE 1284 driver. The host writes ASCII
bytes over USB CDC serial; the Arduino sketch handles per-byte
Centronics handshake on real GPIO pins.

**Pin map** (Centronics 36-pin, printer side — Juki 2200 connector):

| Centronics pin | Signal     | Direction | Arduino role |
|---------------:|------------|-----------|--------------|
| 1              | /STROBE    | OUT       | Pulse low ≥0.5 µs after data stable |
| 2..9           | D0..D7     | OUT       | Drive bus, hold ≥0.5 µs around /STROBE |
| 10             | /ACK       | IN        | Optional — printer pulses after ACK |
| 11             | BUSY       | IN        | Wait for low before next /STROBE |
| 12             | PE (Paper End) | IN    | Surface as USB-side error |
| 13             | SELECT     | IN        | Verify printer online |
| 14             | /AUTO_FEED | OUT       | Tie high (or expose as flag) |
| 15             | /ERROR     | IN        | Surface as USB-side error |
| 16             | /INIT      | OUT       | Pulse low to reset typewriter |
| 17             | /SLCT_IN   | OUT       | Tie low to keep printer selected |
| 19..30         | GND        | —         | Common ground |

**Arduino sketch outline** (sketch_juki2200.ino):

```cpp
const int D[8] = {2, 3, 4, 5, 6, 7, 8, 9};
const int STROBE = 10, BUSY = 11, ACK = 12, SELECT = A0, ERR = A1, INIT = A2;

void setup() {
  Serial.begin(115200);
  for (int i = 0; i < 8; i++) pinMode(D[i], OUTPUT);
  pinMode(STROBE, OUTPUT); digitalWrite(STROBE, HIGH);
  pinMode(INIT, OUTPUT);   digitalWrite(INIT, HIGH);
  pinMode(BUSY, INPUT_PULLUP);
  pinMode(ACK, INPUT_PULLUP);
  pinMode(SELECT, INPUT_PULLUP);
  pinMode(ERR, INPUT_PULLUP);
  // Pulse /INIT briefly to reset printer
  digitalWrite(INIT, LOW); delayMicroseconds(50); digitalWrite(INIT, HIGH);
}

void writeByte(uint8_t b) {
  while (digitalRead(BUSY) == HIGH) { /* spin until printer is ready */ }
  for (int i = 0; i < 8; i++) digitalWrite(D[i], (b >> i) & 1);
  delayMicroseconds(2);                  // setup time
  digitalWrite(STROBE, LOW);
  delayMicroseconds(2);                  // /STROBE pulse width
  digitalWrite(STROBE, HIGH);
  delayMicroseconds(2);                  // hold time
}

void loop() {
  if (Serial.available()) writeByte(Serial.read());
}
```

**Host-side wiring** in `claude-teletype`:

- New driver class `ArduinoBridgeDriver` in `printer.py` that opens
  `/dev/cu.usbmodem*` (or `/dev/ttyACM0` on Linux) at 115200 baud and
  writes raw bytes.
- New profile `juki-2200-arduino` (or extend `juki-2200` with a
  `transport` field) that selects this driver instead of CUPS / pyusb.
- Setup screen gains an "Arduino bridge" connection method when a
  matching serial port is detected (USB VID/PID of common Arduino chips
  — 0x2341 official, 0x1B4F Sparkfun, 0x239A Adafruit, etc.).
- Existing pacing layer (`pacer.py`) feeds the Arduino driver
  byte-by-byte; the Arduino's per-byte BUSY-wait loop provides true
  spec handshake.

This path is highly deterministic, debuggable, and reusable for any
Centronics device — but requires a small amount of soldering /
breadboarding and an Arduino sketch.

### Path C (last resort): CH341 vendor-specific raw parallel mode

If the new adapter turns out to be another cheap CH341 and Path B is not
yet ready: drive the CH341 alt 2 vendor-specific interface via control
transfers documented in the Linux `ch341.ko` source. Higher complexity,
chip-specific, and brittle; only worth attempting if Paths A and B are
blocked.

## Profile refinements (juki-2200)

Once a working byte path exists (Path A success or Path B prototype),
test these escape sequences in this order to refine
`profiles.BUILTIN_PROFILES["juki-2200"]`:

1. `ESC SUB I` (full Juki/Diablo reset) — does the platen advance?
2. `ESC FS NL` (Diablo HMI reset) — do margins reset to 0/132?
3. `ESC RS 9` (1/6" line spacing) — do successive `\r\n` pairs land on
   correct vertical positions?
4. `ESC Q` (fixed pitch) — does proportional-spacing get disabled?
5. Form feed (`0x0C`) — does the platen eject a page (likely no on a
   typewriter; expect to leave `formfeed_on_close=False`).
6. Margin set: try `ESC 9` and `ESC 0` for left/right margins at current
   column position (Diablo 630 convention).

For each, capture the printed output to inform a definitive
`juki-2200` profile dataclass.
